================================================================================
CLOUDFLARE PAGES – BUILD COMMAND FEHLT? (Supabase-Login geht dann nicht)
================================================================================
Wo Sie den Build-Befehl eintragen (genauer Klickpfad):

  1) https://dash.cloudflare.com anmelden
  2) Links: **Workers & Pages**
  3) Unter der Überschrift **Pages** Ihr Projekt anklicken (Name Ihrer Website / GitHub-Repo – kein „Worker“ aus Versehen).
  4) Oben die Registerkarte **Settings** (nicht „Deployments“).
  5) In der **linken Spalte** unter Settings nach einem der folgenden Punkte suchen (UI-Versionen weichen ab):
        • **Build**  oder  **Builds**  oder  **Builds & deployments**
     Dann: **Configure build**, **Edit configuration**, **Production** bearbeiten o. ä.
  6) Im Formular ausfüllen:
        **Framework preset:** None  (oder „None“ / leer – kein Astro/React-Preset nötig)
        **Build command:**        npm run build
        **Build output directory:**  /
             (ein Schrägstrich = Projektroot mit index.html; danach liegen index.html und js/ auf der Site)
        **Root directory:**       leer lassen  (nur bei Monorepo Unterordner, z. B. „website“)
  7) **Save** / **Speichern**
  8) Tab **Deployments** → beim letzten Build **⋯** → **Retry deployment**

Ohne Schritt 6 wird **js/config.js** beim Deploy nicht erzeugt – Environment variables greifen dann nicht im Browser.

================================================================================

Empfehlung: Cloudflare Pages oder Netlify (statisches Hosting, kostenloser Einstieg, CDN, weniger Konfiguration als klassisches Webspace-FTP).

Tägliche Arbeit (GitHub + Cloudflare verbunden):
  Änderungen nur im Git-Repository (Cursor oder github.com) committen und pushen — kein separater Upload zu Cloudflare. Jeder Push auf den verbundenen Branch startet automatisch einen neuen Deploy.

Dieser Ordner ist dafür vorbereitet:
  _headers      → Sicherheits-Header (Cloudflare Pages + Netlify)
  vercel.json   → gleiche Header für Vercel
  netlify.toml  → Publish-Root für Netlify (optional, oft in der UI gesetzt)
  robots.txt    → Suchmaschinen
  404.html      → Fehlerseite
  favicon.svg   → Tab-Icon
  .htaccess     → nur Apache (z. B. InfinityFree); bei Pages/Netlify unwirksam

Deployment (kurz):
1) Cloudflare Dashboard → Workers & Pages → Create → Pages → Upload assets ODER Git verbinden.
2) Build-Einstellungen (wichtig für Supabase-Login):
     Build command: npm run build
     Build output directory: /   (= Root mit index.html)
   Ohne diesen Build wird js/config.js nicht erzeugt – dann funktioniert Anmelden/Admin/Mein Abo nicht, auch wenn Environment variables gesetzt sind.
3) Eigene Domain anbinden (DNS bei Cloudflare oder CNAME laut Anleitung).
4) Supabase: Project Settings → Authentication → URL Configuration
   → Site URL = Ihre Live-URL (https://…)
   → Redirect URLs: dieselbe URL + ggf. Preview-URLs von Pages + passwort-neu.html
     (z. B. https://www.ihre-domain.at/** und die genaue URL …/passwort-neu.html).
   Ausführlich: SUPABASE-EINRICHTUNG.md im Website-Ordner.
5) Mit GitHub + Cloudflare Pages: Im Cloudflare-Projekt unter „Environment variables“ eintragen:
     DAHOAM_SUPABASE_URL, DAHOAM_SUPABASE_ANON_KEY (optional DAHOAM_BAUFPLAN_EMAIL_TO).
     Build command: npm run build  (erzeugt js/config.js beim Deploy; die Datei bleibt aus .gitignore raus).
   Nur ZIP-Upload ohne Git: js/config.js aus js/config.example.js anlegen und mit hochladen.

Alternative InfinityFree: Dateien in den genau zu Ihrer Domain gehörenden …/htdocs/-Ordner legen (Client Area → Domains → Manage). Keine Ordner/Dateinamen mit „chat“ oder Ordner „includes“ in der URL.
